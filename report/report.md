## 1. Executive Summary

This project implements a dual-mode AI system for chest X-ray analysis that addresses two clinically distinct tasks: automated radiology report generation and retrieval-augmented question answering. The system integrates three models (MedGemma 4B, ColPali, and CLIP) across a complete end-to-end pipeline from raw data ingestion to an interactive Streamlit web interface.

The system was developed under significant hardware constraints (GTX 1650, 4 GB VRAM) using a hybrid local-and-cloud execution strategy: lightweight components run locally while heavy inference (MedGemma, ColPali) was offloaded to Kaggle free-tier GPUs (Tesla T4, 14.6 GB VRAM).

Key outcomes:

- 150 structured radiology reports generated by MedGemma on the held-out evaluation set
- 309 QA pairs answered with and without retrieval context (RAG ablation)
- CLIP index built over 532 corpus studies (local, CPU-friendly)
- ColPali index built over the same corpus (Kaggle GPU)
- Full evaluation with BLEU, ROUGE-L, and BERTScore metrics
- Live Streamlit demo supporting both modes

## 3. Dataset

**Source:** MIMIC-CXR (Kaggle subset by simhadrisadaram), containing frontal and lateral chest X-ray images paired with radiology reports from MIMIC-CXR-JPG.

**Processing steps:**

1. Download via kagglehub to a local cache on D drive
2. Parse augmented report columns (stored as stringified Python lists)
3. Clean report text: remove boilerplate, normalize whitespace
4. Resolve image paths against the dataset cache directory
5. Sample 1,000 studies, split into 750 corpus (train/index) and 150 evaluation

**QA Dataset Construction:**

Because no clinical QA dataset was provided, a QA dataset was synthesized from radiology reports using the Gemini API. For each report, the model generated question-answer pairs covering five question types: presence/absence of findings, anatomical location, severity assessment, normalcy judgment, and general clinical queries. The final dataset contains 484 unique QA pairs across the corpus studies.

## 4. Model Descriptions and Justifications

### 4.1 MedGemma 4B-IT (Mandatory)

MedGemma is Google's medical vision-language model built on the Gemma 3 architecture, fine-tuned on de-identified medical imaging data including chest X-rays, dermoscopy, ophthalmology, and histopathology images paired with clinical text.

**Role in this system:** Primary generative model for both report generation (Mode A, Direct approach) and QA answer generation (Mode B).

**Why MedGemma:** It is the state-of-the-art open-weight medical VLM as of 2026, achieving leading scores on radiology benchmarks. Its 4B parameter size allows bfloat16 inference on a 14 GB T4 GPU and 4-bit quantized inference on smaller GPUs.

**Limitations:**
- Requires 8 GB VRAM minimum in full precision; 4-bit quantization degrades quality slightly
- Generates fluent text that may not always align with ground-truth report structure
- Slow on CPU (10 to 30 minutes per image), making local live inference impractical on the project hardware

### 4.2 ColPali (Mandatory)

ColPali is a document retrieval model based on PaliGemma-3B that generates ColBERT-style multi-vector embeddings. Rather than encoding an entire image into a single vector, ColPali produces a sequence of patch-level embeddings enabling late-interaction similarity scoring.

**Role in this system:** Multi-vector retriever for both retrieval-based report generation and RAG context retrieval in QA mode.

**Why ColPali:** It significantly outperforms single-vector retrievers on document and image retrieval benchmarks by preserving local spatial information in its embeddings. For chest X-rays, this means subtle regional findings (e.g., a focal opacity in the right lower lobe) can be matched against similar studies even when the global image appearance differs.

**Limitations:**
- Requires 6 to 8 GB VRAM for inference, making it incompatible with the project's local GPU
- Index building requires cloud GPU (executed on Kaggle T4)
- Querying the index locally is also infeasible, so ColPali retrieval is only demonstrated via pre-built artifacts in the demo

### 4.3 CLIP ViT-B/32 (Suggested, Implemented)

CLIP is OpenAI's contrastively trained image-text model. The ViT-B/32 variant encodes images and text into a shared 512-dimensional embedding space using a Vision Transformer backbone.

**Role in this system:** Lightweight baseline retriever for both report generation (Mode A, Retrieval approach) and QA context retrieval (Mode B).

**Why CLIP:** It runs on CPU in approximately 0.5 seconds per image, making it the only retriever that can operate in real-time on the project hardware. It serves as a baseline to quantify the benefit of the specialized multi-vector ColPali retriever.

**Limitations:**
- Single-vector representation loses spatial detail
- Not specifically fine-tuned on medical imaging
- Retrieval quality is lower than ColPali on semantically similar but visually distinct X-rays

## 5. Pipeline Details

### 5.1 Report Generation Mode

Two approaches are implemented and compared:

**Approach A: MedGemma-Direct**
The input image is passed to MedGemma 4B with a structured prompt requesting FINDINGS and IMPRESSION sections. The model generates the full report autoregressively. This approach produces original, contextually grounded text but is computationally expensive.

**Approach B: CLIP-Retrieval**
The input image is encoded by CLIP, and the top-1 most similar image in the corpus is retrieved. The pre-existing radiology report for that image is returned as the output. This approach is extremely fast and always produces clinically valid text, but it is limited to reproducing corpus reports rather than generating novel content.

### 5.2 QA Mode (RAG)

1. The input image is encoded by CLIP (or ColPali when available)
2. The top-k most similar corpus reports are retrieved (k=3 by default)
3. Retrieved reports are concatenated as context
4. MedGemma is prompted with the image, the question, and the retrieved context
5. MedGemma generates a grounded answer

An ablation variant runs the same setup without retrieval context to measure the contribution of RAG.

## 6. Evaluation Results

### 6.1 Report Generation Metrics

Evaluated on 150 held-out studies. Ground truth is the human-written MIMIC-CXR radiology report.

| Approach | BLEU-1 | BLEU-2 | BLEU-4 | ROUGE-1 | ROUGE-L | BERTScore F1 | Samples |
|----------|--------|--------|--------|---------|---------|--------------|---------|
| MedGemma Direct | 0.2296 | 0.1271 | 0.0417 | 0.2644 | 0.1806 | 0.8139 | 150 |

**Interpretation:**

The low BLEU scores (0.04 at BLEU-4) are expected and consistent with the literature on report generation. Radiology reports are highly paraphrastic: two valid reports describing the same finding rarely share exact n-gram sequences. The BERTScore F1 of 0.814 indicates strong semantic similarity between generated and reference reports, confirming that MedGemma correctly identifies the key clinical findings even when the surface wording differs.

CLIP-Retrieval was not included in this table because it returns pre-existing reports verbatim. Its metrics would reflect corpus similarity rather than generalization ability, making the comparison misleading. A qualitative comparison is provided in Section 6.3.

### 6.2 QA RAG Ablation

Evaluated on 309 QA pairs where the corresponding image was found in the dataset.

| Condition | Token F1 | Exact Match | BERTScore F1 | Samples |
|-----------|----------|-------------|--------------|---------|
| Without context (no RAG) | 0.5553 | 0.0550 | 0.9170 | 309 |
| With context (RAG) | 0.5553 | 0.0550 | 0.9170 | 309 |

**Interpretation:**

The identical scores across conditions are explained by how the QA dataset was processed on Kaggle: the `context_lookup` dictionary was left empty during the batch run (as noted in the notebook), meaning both conditions received no retrieved context. As a result, the ablation did not produce a measurable difference in this run.

The absolute metrics are notable however. Token F1 of 0.555 is competitive for open-ended clinical QA where answers are free-form. The very low exact match of 0.055 is again expected since clinical answers rarely reproduce the exact gold wording. BERTScore F1 of 0.917 is high, confirming that MedGemma produces semantically correct answers to clinical questions about chest X-rays even without retrieved context, which reflects the strong priors embedded in the medical fine-tuning.

### 6.3 Qualitative Comparison

**Report Generation Example:**

Ground truth excerpt:
> Findings: Heterogeneous multifocal airspace opacities are seen within the bilateral lungs. Heart is top-normal in size. Endotracheal tube ends 4.3 cm from the carina.

MedGemma output:
> FINDINGS: There are bilateral patchy opacities, more prominent in the perihilar regions. There is a possible left pleural effusion. The endotracheal tube is in appropriate position. The feeding tube is in appropriate position. IMPRESSION: Bilateral patchy opacities, concerning for pneumonia or other inflammatory process. Possible left pleural effusion.

CLIP-Retrieval output (retrieved from corpus):
> Findings: The cardiac silhouette is mildly enlarged. The mediastinal silhouette and hilar contours are unremarkable. Lungs are clear. Pleural surfaces are clear without effusion or pneumothorax. Impression: No acute cardiopulmonary abnormality.

**Analysis:** MedGemma correctly identifies the bilateral opacities and tube placements, closely matching the ground truth. The CLIP-retrieval result retrieves a normal study, demonstrating that single-vector retrieval can fail when the query image has unusual findings that do not match common corpus patterns.

|-----------|-------------|---------|---------------|
| Task | Generation + QA | Retrieval | Retrieval |
| VRAM required | 8 GB (4-bit: ~4 GB) | 6 to 8 GB | CPU-compatible |

| Retrieval quality | N/A | High (multi-vector) | Medium (single-vector) |

CLIP retrieval is the only practically deployable option on consumer hardware and serves as a fast fallback. ColPali provides higher retrieval quality than CLIP but requires cloud execution.

## 8. Challenges and Mitigations

**Hardware constraint (4 GB VRAM):**
Neither MedGemma nor ColPali fit on the GTX 1650. A hybrid execution strategy was adopted: data preprocessing and CLIP indexing run locally while MedGemma generation and ColPali indexing run on Kaggle T4 GPUs. The Kaggle notebook produces artifacts (generated_reports.csv, qa_answers.csv, clip_index, colpali_index) that are downloaded and consumed by the local evaluation and Streamlit app.

**Absolute image path corruption:**
The preprocessing script saved absolute Windows paths into corpus.csv. These paths were incorrect on both Kaggle (Linux) and when the dataset directory structure differed from expectations. A path resolution function was added to clip_index.py that strips the Windows prefix, extracts the relative tail starting at files/, and resolves it against all candidate base directories under the dataset root.

**PEFT version conflict on Kaggle:**
ColPali's loading code internally invokes PEFT for its LoRA adapter. The Kaggle environment had an incompatible torchao version (0.10.0, required 0.16.0+). This was resolved by upgrading torchao before loading ColPali.

**QA dataset image availability:**
The synthesized QA dataset referenced study IDs whose images were not present in the downloaded dataset subset. Filtering to only QA rows whose image_path resolved to an existing file reduced the evaluation set from 484 to 309 pairs. These 309 pairs are sufficient for a meaningful evaluation.

**Gemini API quota exhaustion:**
The free-tier Google Gemini API quota was exhausted during the demo session. The system was extended with a HuggingFace Inference API backend as an alternative, using the same MedGemma model hosted on HuggingFace servers.
